# Migrating from OpenClaw to picoclaw

> Guide for converting existing OpenClaw projects to the picoclaw runtime.
>
> Korean version: [ko/MIGRATION.md](ko/MIGRATION.md)

---

## Prerequisites

- claw-farm v0.3.0+
- Existing OpenClaw project registered with claw-farm

---

## When to migrate

- You want to reduce resource usage (75x less memory per instance)
- You don't need OpenClaw-specific features (plugins, marketplace, browser automation)
- You're running many user instances on limited hardware

## When NOT to migrate

- You need OpenClaw plugins or ClawHub marketplace
- You use mem0 processor (not yet supported with picoclaw)
- You need browser automation features
- You're running a single instance and resources aren't a concern

---

## Migration steps (single-instance)

```bash
# 1. Stop the current project
claw-farm down my-project

# 2. Back up your data
cp -r openclaw/ openclaw-backup/

# 3. Run the migration command
claw-farm migrate-runtime my-project --to picoclaw

# 4. Verify and start
claw-farm up my-project
```

If `migrate-runtime` is not yet available (v0.3.0), manual steps:

```bash
# 1. Stop and back up
claw-farm down my-project
cp -r openclaw/ openclaw-backup/

# 2. Create picoclaw directory structure
mkdir -p picoclaw/workspace/{memory,sessions,state,skills}

# 3. Copy data files
cp openclaw/workspace/SOUL.md picoclaw/workspace/
cp openclaw/workspace/MEMORY.md picoclaw/workspace/memory/
cp openclaw/workspace/USER.md picoclaw/workspace/ 2>/dev/null
cp -r openclaw/workspace/skills/* picoclaw/workspace/skills/ 2>/dev/null

# 4. Update .claw-farm.json
# Change "runtime" to "picoclaw" (or add if missing)
# Optionally add "proxyMode": "shared"

# 5. Regenerate compose and config
claw-farm upgrade my-project

# 6. Start with new runtime
claw-farm up my-project
```

## Migration steps (multi-instance)

```bash
# 1. Stop all instances
claw-farm down my-project

# 2. Back up
cp -r instances/ instances-backup/
cp -r template/ template-backup/

# 3. For each instance, convert directory structure:
for user in instances/*/; do
  uid=$(basename "$user")
  mkdir -p "instances/$uid/picoclaw/workspace/{memory,sessions,state,skills}"
  cp "instances/$uid/openclaw/workspace/SOUL.md" "instances/$uid/picoclaw/workspace/"
  cp "instances/$uid/openclaw/workspace/MEMORY.md" "instances/$uid/picoclaw/workspace/memory/"
  cp "instances/$uid/openclaw/workspace/USER.md" "instances/$uid/picoclaw/workspace/" 2>/dev/null
  cp -r "instances/$uid/openclaw/workspace/skills/"* "instances/$uid/picoclaw/workspace/skills/" 2>/dev/null
done

# 4. Update template config
mv template/config/openclaw.json template/config/openclaw.json.backup
# config.json will be regenerated by upgrade

# 5. Update .claw-farm.json
# Set "runtime": "picoclaw"

# 6. Regenerate all
claw-farm upgrade my-project

# 7. Start
claw-farm up my-project
```

## What gets migrated

| Data | Source (OpenClaw) | Destination (picoclaw) |
|------|-------------------|----------------------|
| SOUL.md | openclaw/workspace/SOUL.md | picoclaw/workspace/SOUL.md |
| MEMORY.md | openclaw/workspace/MEMORY.md | picoclaw/workspace/memory/MEMORY.md |
| USER.md | openclaw/workspace/USER.md | picoclaw/workspace/USER.md |
| Skills | openclaw/workspace/skills/ | picoclaw/workspace/skills/ |
| Session logs | openclaw/sessions/*.jsonl | Not migrated (format differs) |
| Config | openclaw.json + policy.yaml | config.json (regenerated) |

## What does NOT migrate

- Session logs (OpenClaw JSONL format differs from picoclaw's)
- OpenClaw-specific config (policy.yaml, controlUi settings)
- Processed memory (Layer 1) — can be rebuilt with `claw-farm memory:rebuild`

---

## Rollback

```bash
# Stop picoclaw
claw-farm down my-project

# Restore OpenClaw data
rm -rf picoclaw/
cp -r openclaw-backup/ openclaw/

# Revert .claw-farm.json (remove "runtime" field or set to "openclaw")
claw-farm upgrade my-project
claw-farm up my-project
```

## proxyMode selection guide

| Scenario | Recommended proxyMode |
|----------|---------------------|
| All users share same LLM API key | `shared` (saves resources) |
| Users have different API keys | `per-instance` |
| Users have sensitive secrets (wallets, trading keys) | `per-instance` |
| Cost-sensitive deployment | `shared` |
| Maximum security isolation | `per-instance` |
